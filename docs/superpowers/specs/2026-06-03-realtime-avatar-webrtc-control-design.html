实时数字人 WebRTC 控制服务设计

日期：2026-06-03

1. 目标

本文档定义一个新的实时数字人控制与播放服务。目标是提供一个可通过浏览器访问的控制页面，让用户连接 Python 实时推理服务，启动文字驱动的数字人合成，并通过 WebRTC 播放音视频同步的结果。

本文档不定义 OBS 集成行为。如果用户希望在 OBS 中采集浏览器页面，那属于使用方式，不属于本系统的功能范围。

2. 已确认范围

MVP 范围内

  • 使用 Python 服务，并基于 aiortc 实现 WebRTC。
  • WebRTC 服务与实时推理模块运行在同一个 Python 进程中。
  • 通过内网 IP 和端口访问，MVP 不做鉴权。
  • 只提供一个页面：GET /avatar/control。
  • 同一时间只允许一个活跃客户端/会话。
  • 控制页面包含：
    • 连接按钮。
    • 文本输入框。
    • 开始合成按钮。
    • 停止合成按钮。
    • 状态展示。
    • 页面内 WebRTC 同步音视频播放区域。
  • 使用 WebSocket 承载信令和会话控制。
  • WebRTC 输出必须同时包含视频轨和音频轨。
  • 文字合成架构按“分段流式合成”设计。
  • MVP 可以先把用户提交的整段文本作为一个分段处理。
  • 麦克风音频输入仅预留扩展，不在 MVP 实现。
  • 没有推理会话运行时，不应产生推理音视频。
  • 空闲状态由页面层显示占位/测试图案。

MVP 范围外

  • OBS 专用 URL 参数或 OBS 采集行为。
  • 鉴权和 token 校验。
  • 公网安全加固。
  • 多客户端观看。
  • 多会话支持。
  • 麦克风实时驱动。
  • 真正按键级别的增量合成。
  • 等待当前分段播放完成后再停止的优雅停止。
  • 将推理服务和 WebRTC 服务拆成独立进程。
  • 本地直接访问云端 /dev/videoX 设备。

3. 系统背景

当前 Duix-Avatar 是任务式合成架构：客户端提交完整音频/视频输入到后端，轮询合成状态，最终得到生成后的视频文件。它不是实时帧流架构。

新的服务职责不同：它需要把实时推理抽象成一个会话，由该会话持续产出音视频同步帧，并把这些帧写入 WebRTC 音视频轨。

4. 高层架构

浏览器
  GET /avatar/control
  WS  /api/session
        |
        v
Python 实时数字人服务
  ├─ HTTP 页面服务
  ├─ WebSocket 会话/信令控制器
  ├─ aiortc PeerConnection 管理器
  ├─ 文本分段器和分段队列
  ├─ 实时推理会话
  └─ WebRTC 音视频轨

服务端同一时间只拥有一个活跃会话。页面先连接服务，连接成功后用户提交文本并启动合成。推理会话随后产生音视频同步数据，并通过已建立的 WebRTC 连接发送到浏览器。

5. HTTP 接口

GET /avatar/control

返回控制页面。

控制页面必须包含：

  • 连接状态。
  • 媒体播放区域。
  • 文本输入区域。
  • 连接按钮。
  • 开始合成按钮。
  • 停止合成按钮。
  • 合成/会话状态。

MVP 不需要也不设计任何 URL 参数。

6. WebSocket 接口

WS /api/session

WebSocket 负责：

  • 占用单个活跃控制会话。
  • 交换 WebRTC offer/answer 信令。
  • 接收开始/停止命令。
  • 发送会话状态事件。
  • 页面关闭、刷新或异常断开时释放会话。

单会话行为

如果已经存在一个活跃 WebSocket 会话，第二个客户端不能成为活跃客户端。服务端可以直接拒绝连接，也可以短暂接受连接并发送类似 session.busy 的错误事件后关闭。

7. WebSocket 消息类型

所有消息都是 JSON 对象，并包含 type 字段。

客户端到服务端

webrtc.offer

用户点击“连接”后，浏览器创建 SDP offer，并通过该消息发送给服务端。

{
  "type": "webrtc.offer",
  "sdp": "...",
  "sdpType": "offer"
}

session.start

根据提交的文本启动合成。

{
  "type": "session.start",
  "text": "要合成的整段文本"
}

服务端必须校验：

  • WebRTC 连接已经存在或正在建立。
  • 文本不能为空。
  • 当前没有正在运行的合成会话。

session.stop

立即停止当前合成会话。

{
  "type": "session.stop"
}

服务端到客户端

webrtc.answer

服务端接受浏览器 offer 后返回 answer。

{
  "type": "webrtc.answer",
  "sdp": "...",
  "sdpType": "answer"
}

session.status

上报会话状态变化。

{
  "type": "session.status",
  "status": "connected"
}

MVP 允许的状态：

  • idle：空闲，未连接。
  • connecting：正在连接。
  • connected：已连接，未合成。
  • starting：正在启动合成。
  • streaming：正在输出音视频。
  • stopping：正在停止合成。
  • error：错误状态。

session.error

上报可恢复或不可恢复错误。

{
  "type": "session.error",
  "code": "TEXT_EMPTY",
  "message": "Text must not be empty"
}

MVP 推荐错误码：

  • SESSION_BUSY：已有活跃会话。
  • INVALID_MESSAGE：消息格式无效。
  • TEXT_EMPTY：文本为空。
  • WEBRTC_NOT_READY：WebRTC 未就绪。
  • SYNTHESIS_ALREADY_RUNNING：已有合成正在运行。
  • INFERENCE_FAILED：推理失败。
  • INTERNAL_ERROR：内部错误。

segment.status

上报分段队列进度。即使 MVP 把整段文本作为一个分段，也应使用分段状态结构。

{
  "type": "segment.status",
  "segmentId": "seg-1",
  "status": "streaming"
}

允许的分段状态：

  • queued：已入队。
  • starting：正在启动。
  • streaming：正在输出。
  • completed：已完成。
  • cancelled：已取消。
  • failed：失败。

8. 控制页面状态机

idle
  ↓ 用户点击连接
connecting
  ↓ WebSocket + WebRTC 连接成功
connected
  ↓ 用户输入文本并点击开始合成
starting
  ↓ 推理开始输出同步音视频
streaming
  ↓ 用户点击停止合成
stopping
  ↓ 推理取消并清空队列
connected
  ↓ 页面关闭、刷新或连接异常
idle

页面不提供手动断开按钮。关闭或刷新页面时，服务端释放活跃会话。

9. 媒体语义

已连接但未合成

页面已连接但尚未开始合成时：

  • 播放区域显示占位/测试图案 UI。
  • 推理模块不得产生伪造的推理音视频。
  • 服务内部可以按实现需要保持 WebRTC track 存在，但这些 track 不能被表达为成功的推理输出。

正在合成

合成开始后：

  • 服务创建推理会话。
  • 推理会话产生音视频同步帧。
  • WebRTC 音频轨和视频轨把这些帧发送到浏览器。
  • 控制页面播放该实时流。

音频要求

MVP 输出必须包含音频和视频。只有视频输出不满足本文档要求。

10. 文本合成设计

即使 MVP 只处理单个分段，整体架构也必须按分段模式设计。

MVP 行为

用户提交整段文本
  ↓
TextSegmenter 输出一个分段
  ↓
SegmentQueue 执行该分段
  ↓
InferenceSession 输出音视频同步结果

后续行为

同一套接口后续应支持：

  • 按中文和英文标点切分文本。
  • 按最大长度切分文本。
  • 多个分段排队。
  • 前一段输出时，后一段提前准备。
  • 上报每个分段的状态。
  • 当前分段播放完成后再停止的优雅停止。

本文档不规划的能力

本文档不设计按键级别的增量合成。系统不应在每次文本输入变化时都重新启动合成。

11. 停止语义

MVP 立即停止

用户点击“停止合成”后：

  • 立即取消活跃推理会话。
  • 清空分段队列。
  • 停止当前合成的音频输出。
  • 页面状态回到 connected。
  • 保留 WebSocket 和 WebRTC 连接，以便用户再次点击开始合成。

后续优雅停止

后续版本可以增加优雅停止：

  • 拒绝新的分段。
  • 允许当前分段播放完成。
  • 然后回到 connected 状态。

12. 推理模块边界

虽然 MVP 中推理和 WebRTC 运行在同一个 Python 进程里，但模块边界应保持清晰。

推荐的内部接口：

TextSegmenter
  输入：文本
  输出：有序分段

SegmentQueue
  输入：分段
  输出：分段生命周期事件

InferenceSession
  输入：分段文本
  输出：音视频同步帧
  控制：start、cancel

MediaTrackBridge
  输入：来自推理模块的音视频帧
  输出：aiortc AudioStreamTrack 和 VideoStreamTrack

推理模块不能阻塞 asyncio 事件循环。重型计算应根据需要放到 executor、worker thread 或独立 process 中执行。

13. 后续麦克风扩展

MVP 不实现麦克风输入，但架构不应阻断后续扩展。

后续麦克风链路：

浏览器麦克风
  ↓ WebRTC audio track 或 WebSocket 音频块
Python 服务
  ↓ 实时推理会话
音视频同步的数字人输出
  ↓ WebRTC 播放

MVP 控制页面可以不显示麦克风控件。如果存在占位 UI，必须明确禁用或隐藏，避免让用户误以为已支持。

14. 部署假设

  • 服务运行在云端或内网主机上。
  • MVP 期间用户通过 http://<ip>:<port>/avatar/control 访问。
  • MVP 不要求 TLS 或鉴权。
  • 公网部署需要后续安全评审，并通常需要 HTTPS。

15. 验收标准

MVP 满足以下条件即可验收：

  1. 用户可以打开 http://<ip>:<port>/avatar/control。
  2. 页面初始显示空闲状态和占位播放区域。
  3. 点击“连接”后建立单个活跃会话和 WebRTC 连接。
  4. 第二个并发客户端不能接管活跃会话。
  5. 输入非空文本并点击“开始合成”后创建合成会话。
  6. 合成运行时，服务通过 WebRTC 发送音视频同步数据。
  7. 页面能够同时播放音频和视频。
  8. 点击“停止合成”后立即取消合成，并回到已连接状态。
  9. 关闭或刷新页面后释放活跃会话。
  10. 内部文本流程使用分段抽象，即使 MVP 只把整段文本作为一个分段。

16. 本文档未解决的开放问题

以下问题留到实施计划或后续 spec 中处理：

  • 具体实时推理模型 API。
  • 推理模块输出的具体音视频帧格式。
  • 目标延迟预算。
  • 浏览器自动播放策略处理细节。
  • 生产环境鉴权。
  • TLS 和域名部署。
  • 多客户端观看。
  • OBS 相关使用说明。
