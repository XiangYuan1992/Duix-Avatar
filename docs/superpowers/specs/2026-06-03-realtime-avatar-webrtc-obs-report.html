核对后报告 Python + aiortc /avatar/control WebSocket 控制

实时数字人 WebRTC 控制服务核对报告

本报告同步我们最终核对后的 specs 信息，用于替换早期“WebRTC OBS 展示页”方向。当前系统范围聚焦于一个控制页面和实时推理音视频播放，不把 OBS 使用方式作为系统设计范围。

最终核对结论： 系统只提供 /avatar/control 控制页面。页面负责连接实时服务、输入文本、开始/停止合成，并播放 WebRTC 同步音视频。OBS 采集方式由使用者自行处理，不纳入本报告和 spec 的功能范围。

1. 背景判断

当前 Duix-Avatar 主流程是离线任务式合成：提交音频和数字人视频，后端异步生成完整视频文件，客户端轮询结果。它不是实时帧流架构。

当前离线链路：
文字 → TTS 生成音频
上传音频 → 音频文件
音频文件 + 数字人视频 → submit 异步合成任务
轮询 query → 获取生成后的视频文件

因此，要满足实时文字/音频驱动口型，需要设计一个新的实时服务。该服务的核心不是改造 OBS 接入，而是让推理模块能够产出同步音视频，并通过 WebRTC 在浏览器页面播放。

2. 最终确认的系统范围

MVP 范围内

  • Python 服务，使用 aiortc。
  • WebRTC 服务和实时推理模块运行在同一个 Python 进程。
  • 内网 IP:端口访问，无鉴权。
  • 只提供一个页面：GET /avatar/control。
  • 单会话单客户端。
  • WebSocket 负责信令和会话控制。
  • 页面播放 WebRTC 同步音频和视频。
  • 页面支持连接、文本输入、开始合成、停止合成、状态展示。
  • 文本合成按分段流式架构设计，MVP 可把整段文本作为一个 segment。
  • 麦克风输入预留，不在 MVP 实现。

MVP 范围外

  • OBS 专用参数或 OBS 采集行为。
  • 公网鉴权、Token、TLS、正式安全策略。
  • 多客户端观看。
  • 多会话并发。
  • 麦克风实时驱动。
  • 逐字输入即触发的增量合成。
  • 优雅停止，即等待当前 segment 播放完成后再停止。
  • 推理服务和 WebRTC 服务拆成独立进程。
  • 本地直接访问云端 /dev/videoX。

3. 核心架构

浏览器
  GET /avatar/control
  WS  /api/session
        |
        v
Python Realtime Avatar Service
  ├─ HTTP 页面服务
  ├─ WebSocket 会话和信令控制器
  ├─ aiortc PeerConnection 管理器
  ├─ 文本分段器和 segment 队列
  ├─ 实时推理 session
  └─ WebRTC audio/video tracks

页面先点击“连接”，建立 WebSocket 会话和 WebRTC 连接。连接成功后，用户输入文本并点击“开始合成”。推理 session 开始输出同步音视频帧，WebRTC tracks 将其传回页面播放。

4. 页面设计

只提供 /avatar/control 页面。该页面既是控制页，也是播放页。

页面控件

  • 连接按钮。
  • 文本输入区域。
  • 开始合成按钮。
  • 停止合成按钮。
  • 连接状态展示。
  • 合成状态展示。
  • 音视频播放区域。

页面约束

  • 不需要额外 URL 参数。
  • 不需要手动断开连接按钮。
  • 页面关闭或刷新时释放服务端会话。
  • 未开始推理时，页面层显示占位或测试图案。
  • 未开始推理时，不伪造推理音视频输出。

5. WebSocket 信令和控制

WebSocket 地址为 WS /api/session。它负责单会话占用、WebRTC offer/answer、开始合成、停止合成、状态推送和异常释放。

客户端 → 服务端：
{ "type": "webrtc.offer", "sdp": "...", "sdpType": "offer" }
{ "type": "session.start", "text": "要合成的整段文本" }
{ "type": "session.stop" }

服务端 → 客户端：
{ "type": "webrtc.answer", "sdp": "...", "sdpType": "answer" }
{ "type": "session.status", "status": "connected" }
{ "type": "session.error", "code": "TEXT_EMPTY", "message": "Text must not be empty" }
{ "type": "segment.status", "segmentId": "seg-1", "status": "streaming" }

6. 控制页状态机

idle
  ↓ 点击连接
connecting
  ↓ WebSocket + WebRTC 连接成功
connected
  ↓ 输入文本 + 点击开始合成
starting
  ↓ 推理开始输出同步音视频
streaming
  ↓ 点击停止合成
stopping
  ↓ 取消推理并清空队列
connected
  ↓ 页面关闭、刷新或连接异常
idle

停止合成只停止当前推理，不断开页面连接。页面关闭或刷新时才释放连接和单会话占用。

7. 媒体输出语义

连接但未合成

  • 页面显示占位或测试图案。
  • 推理模块不输出伪造音视频。
  • 服务可保留内部 WebRTC tracks，但不能把它们当成真实推理输出。

合成中

  • 创建推理 session。
  • 推理 session 输出同步音频和视频帧。
  • WebRTC audio/video tracks 将数据送到浏览器。
  • 页面播放同步音视频。
音视频要求：MVP 必须同时输出音频和视频。仅视频输出不满足核对后的 spec。

8. 文本合成设计

文本合成必须按 segment 架构设计，即使 MVP 先把整段文本当成一个 segment。

MVP：
提交整段文本
  ↓
TextSegmenter 生成一个 segment
  ↓
SegmentQueue 执行该 segment
  ↓
InferenceSession 输出同步音视频

后续：
按中文/英文标点切分
按最大长度切分
支持多个 segment 队列
支持前一段播放时准备后一段
支持每段状态回调
支持优雅停止

逐字输入变化立即触发重新合成不在当前设计范围内。

9. 停止语义

MVP 使用立即停止：

  • 立即取消当前 inference session。
  • 清空 segment 队列。
  • 停止当前音频输出。
  • UI 回到 connected 状态。
  • WebSocket 和 WebRTC 连接保留，可再次点击开始合成。

后续可扩展优雅停止：不再接收新 segment，让当前 segment 播放完成后停止。

10. 内部模块边界

TextSegmenter
  输入：文本
  输出：有序 segments

SegmentQueue
  输入：segments
  输出：segment 生命周期事件

InferenceSession
  输入：segment 文本
  输出：同步音视频帧
  控制：start、cancel

MediaTrackBridge
  输入：推理产生的音视频帧
  输出：aiortc AudioStreamTrack 和 VideoStreamTrack

推理模块不能阻塞 asyncio 事件循环。重计算应放到 executor、worker thread 或独立 worker process 中执行，但 MVP 的服务进程仍保持单进程架构边界。

11. 麦克风输入预留

MVP 不实现麦克风实时驱动，但架构需要为后续扩展保留空间。

未来麦克风链路：
浏览器麦克风
  ↓ WebRTC audio track 或 WebSocket audio chunks
Python 服务
  ↓ 实时推理 session
同步数字人音视频输出
  ↓ WebRTC 页面播放

MVP 页面可以不显示麦克风控件。如果出现占位控件，必须明确禁用或隐藏，避免误认为已支持。

12. 验收标准

  1. 用户可以打开 http://<ip>:<port>/avatar/control。
  2. 页面初始显示 idle 状态和占位播放区域。
  3. 点击连接后建立单活动会话和 WebRTC 连接。
  4. 第二个并发客户端不能接管当前活动会话。
  5. 输入非空文本并点击开始合成后创建 synthesis session。
  6. 合成运行时服务通过 WebRTC 发送同步音频和视频。
  7. 页面能同时播放音频和视频。
  8. 点击停止后立即取消合成，并回到 connected 状态。
  9. 关闭或刷新页面后释放活动会话。
  10. 内部文本流程使用 segment 抽象，即使 MVP 只生成一个 segment。

13. 尚未纳入本次核对的开放问题

  • 真实实时推理模型 API。
  • 推理模块输出的音频和视频帧格式。
  • 目标延迟预算。
  • 浏览器自动播放策略处理。
  • 生产环境鉴权。
  • TLS、域名和公网部署。
  • 多客户端观看和多会话。
总结：核对后的方向是一个面向实时推理的 WebRTC 控制服务，而不是 OBS 集成服务。系统只保证 /avatar/control 页面能够连接、控制合成并播放同步音视频。
